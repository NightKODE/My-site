GIT - распределенная система управления версиями.

На каждом компьютере хранится полная версия репозитория, что позволяет локально работать над продуктом, даже без интернета.
Lesson 1.
Для чего нужен GIT.
  • Позволяет сохранить разные версии проекта.
  • Позволяет быстро переходить к разным версиям проекта.
  • Позволяет откатиться к предыдущим версиям приложения.
  • Упрощает и улучшает процесс командной работы над одним проектом.
  • Позволяет осуществлять разработку различных функций в проекте в отдельной ветке независимо от всех остальных разработчиков.
Lesson 2.
GIT и GitHub.
GIT
Распределенная система контроля версий. Локальная работа над продуктом: создание веток, ответвлений, версий и т.д.
GitHub
Сервис хостинга Git репозиториев. Хранит основную версию продукта для разработчиков.
Lesson 3.
Рабочие области GIT.
  • Working directory - рабочая директория.

    Видимая область проекта в которой содержатся файлы (папка проекта).

  • Staging area - индекс.

    В эту область добавляются файлы перед сохранением.

    git status - показывает изменения в индексе которые были подготовлены для сохранения (коммита).

  • Repository - репозиторий (папка.git).

    Здесь сохраняются все версии файлов проекта.

    Так же тут сохраняются коммиты (указатели на определенные версии проекта).

    На примере папки Objekts (в ней сохраняются все объекты репозитория).

    Папки появляются после создания объектов gitом.

    Файлы с длинными названиями (их названия основаны на хешах git-объектов).

    При изменении файла, в репозитории (папка Objects) сохранится два разных файла (первая версия и вторая).

Lesson 4.
Рабочий процесс GIT.
  • 1. Создается папка с файлами - проект.
    В нее добавлять файлы можно традиционным и не только способами.
  • 2. В терминале прописывается команда git init.
    Она нужно для инициализации репозитория GIT и создания скрытой папки.git.
  • 3. В индекс файлы добавляются командой - git add “name”
    После команды прописывается имя файла, если поставить точку, то добавятся все модифицированные файлы и созданные.
  • 4. После добавления изменений в индекс можно сохранить текущую версию проекта в репозиторий командой - git commit -m "messege".
  • 5. Командой git checkout можно перейти к определенной версии проекта.
    Из репозитория в рабочую директорию перезаписываются файлы, либо добавляются, либо удаляются, тем самым проект будет состоять из нужных версий файлов.
Lesson 5.
Статусы отслеживания файллов.
  • Untracked - неотслеживаемый.

    Новые файлы, добавленные в рабочую директорию имеют статус "неотслеживаемые” т.к. еще не находятся в репозитории. Этот статус имеют все новые файлы изначально.

  • Staged - подготовленный.

    Файлы, добавленные в индекс для сохранения командой git add, имеют статус “подготовленные”

  • Unmpdified - немодифицированный.

    Файлы, сохраненные в репозиторий командой
    git commit, имеют статус “немодифицированный”

  • Modified - модифицированный.

    Файлы (находящиеся в РД, индексе и репозитории), взятые для каких-либо изменений, имеют статус модифицированные. Они могут в дальнейшем иметь статус “подготовленные” и “немодифицированные”.

Lesson 6.
Объекты GIT.
  • Commit - указатель на дерево (ссылка).
  • Blob - файл.
  • Tree - папка.
Lesson 7.
Удаленный репозиторий GitHub.
Удаленный репозиторий позволяет хранить проект удаленно.

Удаленные репозитории бывают публичные и частные. публичные репозитории могут скачивать все без ограничений, но загружать изменения в него только с личным доступом. Личный доступ дает админ репозитория ( владелец на GitHub ). С частным репозиторием работать можно только с личным доступом.
Как работать с удаленным репозиторием.
  • 1. Сначала уделенный репозиторий скачивается на компьютер командой git clone "url".

    При скачивании git создает связь между локальным и удаленным репозиториями, при этом уделенному репозиторию задается имя по умолчанию - origin

  • Далее необходимо подключить удаленный репозиторий к локальному командой
    git remote add origin "url" ( имя удаленного репозитория заданное на GItHub при его создании ).

    После этого на удаленный репозиторий можно загружать какие-лмбо изменения, но связь с ветками еще не устанавлена, это нужно делать далее.

  • Теперь локальную ветку (например “new-feature”) необходимо связать с удаленной веткой (например “main”, как правило связывают ветки с одинаковым названием) командой git push –u origin "branch".

    После создания связи между ветками загружать обновления можно обычной командой git push.

  • После проделанных действий загружать и скачивать какие-либо изменения в локальной ветке можно командами git push и git pull соответственно.
Lesson 8.
Команды GIT.
Основные.
  • git init - cоздание репозитория
  • git status - отображает текущее состояние репозитория
  • git add "file" - добавляет файлы в индекс (поставив точку вместо имени файла добавятся все измененные файлы)
  • git commit –m "message" - создание коммита с записью изменений в репозиторий
  • git log - просмотр истории изменений
  • git checkout "commit hash" - переход в определенную версию проекта по хэшу коммита
Ветвление.
  • git checkout "branch name" - переход на определенную версию проекта по названию ветки
  • git branch "branch name" - создание ветки
  • git checkout –b "branch name" - создание ветки и переход в нее
  • git branch - отображение существующих веток
  • git branch –d "branch name" - удаление ветки ( текущую ветку удалить нельзя )
  • git branch -m "name" - переименование текущей ветки имя.
  • git merge –m "message" "feature branch name" - слияние другой ветки в текущую
Удаленный репозиторий.
  • git clone "url" - клонирование удаленного репозитория к себе на компьютер.
  • git pull - загрузка и применение изменений с удаленной ветки в локальную.
  • git push - загрузка изменений из локальной ветки в ветку удаленного репозитория.
  • git branch –a - отображает все ветки, включая те, которые находятся в удаленных репозиториях.
  • git remote add origin "url" - подключение удаленного репозитория.
  • git push –u origin "branch" - загрузка изменения из локальной ветки в удаленную с созданием связи между ними.
  • git push –u origin "branch" - загрузка изменения из локальной ветки в удаленную с созданием связи между ними. После создания связи между ветками загружать обновления можно обычной командой git push.
Прочие.
  • git –version - проверка версии
  • git config --global user.name “...” - установка имени юзера
  • git config --global user.mail “...” - установка почты юзера
  • git config --list - проверить настройки git
  • git cat-file -t "name" - чтение типа файла.
  • git cat-file -p "name" - чтение содержимого файла.
  • git remote - проверка подключения удаленного репозитория
  • git remote –v - проверка связи между локальным и удаленным репозиториями
  • git branch -vv - список и состояние всех локальных и удаленных веток (только данные, которые обновлены с удаленного сервера).
Lesson 9.
Команды для консоли.
  • cd ~/Desktop - переход в папку для работы в ней (тильда обозначает корневую папку юзера/рабочий стол)
  • cd .. - выход в родительскую папку (на уровень выше )
  • mkdir my-project - создание новой папки (имя папки)
  • ls - проверка наличия файлов в папке
  • ls –Force - отобразить скрытые папки в папке
  • echo "Some text" > file.txt - создание файла (текст в самом файле, название файла и его расширение )
  • cat file.txt - чтение файла
  • rm file.txt - удаление файла
